Основы чистого кода: убираем if/else-ад с помощью функций

Вы наверняка знаете про early return паттерн. Но его не всегда понятно, как применить. Часто бывает так, что вы пишите некоторую последовательность инструкций так, что каждая последующая должна выполняться, только если предыдущая закончилась без ошибок:

func calculateSomething(x, y int){
{ // не основной блок внутри функции, просто заносим аргументы вызова в лог файл
  f, err := os.OpenFile("calculateSomething.log", os.O_CREATE|os.O_RDWR|os.O_TRUNC, l.opts.FilePerms)
  if err == nil { // заходим сюда только если нет ошибки
     defer f.Close()
     if _, err := f.Write(fmt.Sprintf("%d, %d", x, y)); err == nil { 
         if err := f.Sync(); err != nil { // выполнить, только если ранее не было ошибки
           log.Printf("failed to fsync file: %s", err)
         }
     } else {
      log.Printf("failed to write file: %s", err)
     }
     
     return f.Close()
 } else {
    log.Printf("failed to open file: %s", err)
 }
}

}

При этом вы не можете сделать early return, потому что функция должна отработать независимо от блока, связанного с файлами. Вы хотите просто залогировать ошибку работы с файлами и продолжить выполнение. Early return здесь просто не сработает.

Первое, о чем вы можете подумать — можно ли вынести проблемный блок в отдельную функцию?

func logCalc(x, y int) error {
  f, err := os.OpenFile("calculateSomething.log", os.O_CREATE|os.O_RDWR|os.O_TRUNC, l.opts.FilePerms)
  if err != nil {
    return fmt.Errorf("failed to open file: %w", err)
  }
  defer f.Close()

  if _, err := f.Write(fmt.Sprintf("%d, %d", x, y)); err != nil { 
    return fmt.Errorf("failed to write file: %w", err)
  } 
  if err := f.Sync(); err != nil {
    return fmt.Errorf("failed to fsync file: %w", err)
  }
     
  return nil
}

Теперь вы можете уверенно сделать возврат из любой точки вспомогательной функции без необходимости создавать лишнюю вложенность.

Теперь основная функция выглядит так:

func calculateSomething(x, y int){
  if err := logCalc(x, y); err != nil {
    log.Printf("file manipulation error: %s", err)
  }

}

Лучше, не правда ли?

Но что если функция logCalc не используется больше нигде? Не стоит выделять в отдельную функцию то, что больше нигде не используется (исключение составляют только те случаи, когда верхнеуровневая функция слишком большая и ради улучшения читаемости её можно разбить на несколько). Можно переписать наше решение так:

func calculateSomething(x, y int){
// вспомогательная анонимная функция
if err := func(x, y int) error {
  f, err := os.OpenFile("calculateSomething.log", os.O_CREATE|os.O_RDWR|os.O_TRUNC, l.opts.FilePerms)
  if err != nil {
    return fmt.Errorf("failed to open file: %w", err)
  }
  defer f.Close()

  if _, err := f.Write(fmt.Sprintf("%d, %d", x, y)); err != nil { 
    return fmt.Errorf("failed to write file: %w", err)
  } 
  if err := f.Sync(); err != nil {
    return fmt.Errorf("failed to fsync file: %w", err)
  }
     
  return nil
}(); err != nil {
    log.Printf("file manipulation error: %s", err)
}

// основная полезная работа функции
...
}

Мы сделали функцию logCalc анонимной и встроили её в calculateSomething . Функция осталась такой же чистой без if/else-ада, при этом мы устранили лишнюю функцию, которая нигде не используется и может вводить в заблуждение других участников проекта.
